# 6. Shared Filesystem (Manila — NFS)

---

## 6-1. Manila 구조

```
Control Plane: Manila (API/관리)
        ↓
Middle Layer: NFS-Ganesha (NFS 프로토콜/인증/격리)
        ↓
Backend Storage: Ceph (실제 데이터 저장)
```

| 구성요소 | 역할 |
|---|---|
| **Manila** | NFS 구성의 Control Plane. 자원 할당/관리 담당 |
| **NFS-Ganesha** | NFS 프로토콜 구현, 인증 처리, Project 간 네트워크 격리 담당 |
| **Ceph** | 실제 데이터 저장 (백엔드 스토리지) |
| **Share** | NFS의 자원 할당 단위 |
| **Export** | Share를 네트워크상에 공유하는 것 |

> NFS-Ganesha는 데이터를 저장하지 않고
> **클라이언트(VM)와 Ceph 사이의 중개/인증/격리** 역할을 담당

---

## 6-2. DHSS (Driver Handles Share Servers)

**DHSS: True (권장)**

```
사용자 요청
        ↓
Manila-share가 NFS-Ganesha 자동 배포
        ↓
백엔드 스토리지(Ceph) 연결
        ↓
Share를 네트워크에 Export
```

- Project마다 **NFS-Ganesha를 독립적으로 배포**
- Project 간 스토리지 **네트워크 격리** 가능
- DHSS는 **Share Type 생성 시 설정하는 옵션값** (별도 설치 아님)

```bash
# DHSS true로 Share Type 생성
openstack share type create \
  --extra-specs driver_handles_share_servers=True \
  default_share_type
```

**DHSS: False**

```
NFS-Ganesha 배포 안됨
        ↓
백엔드 스토리지와 직접 통신
        ↓
Project 간 격리 안됨 ⚠️
```

---

## 6-3. DHSS가 중요한 이유

```
하나의 스토리지 서버를 여러 Project가 공유할 때
→ Project 간 보안/격리가 필요
→ NFS-Ganesha가 네트워크 격리 역할 담당

DHSS: True  → Ganesha 배포 → Project별 격리 ✅
DHSS: False → Ganesha 없음 → Project별 격리 ❌
```

## 6-4. Ganesha & LDAP 연동

**Ganesha를 LDAP에 등록하면 NFS 사용이 편리해짐**

```
LDAP 미연동 시
→ NFS 접근 시 매번 수동 인증 필요
→ 사용자/권한 관리 복잡

LDAP 연동 시
→ LDAP 계정으로 NFS 자동 인증
→ 중앙에서 사용자/권한 통합 관리
→ 여러 VM에서 동일한 LDAP 계정으로 NFS 접근 가능 ✅
```

**연동 구조**

```
VM (NFS 접근 요청)
        ↓
NFS-Ganesha
        ↓ 인증 요청
LDAP 서버 (사용자 인증/권한 확인)
        ↓ 인증 성공
Ceph 백엔드 스토리지 접근 허용
```

**장점**

| 항목 | 설명 |
|---|---|
| **중앙 인증** | LDAP 계정으로 통합 관리 |
| **편의성** | 별도 keyring 없이 LDAP 계정으로 접근 |
| **보안** | LDAP 기반 접근 제어 |

## 6-5. Ceph 백엔드 연동

**cephx 프로토콜**
- Manila가 Ceph를 백엔드 스토리지로 사용할 때 **인증 프로토콜**
- 인증 키값은 **keyring 파일**에 저장됨

**Keyring 파일 위치**

```
Controller Node
└── /etc/ceph/ceph.client.manila.keyring
    → Manila가 Ceph에 접속할 때 사용하는 인증 키
    → Ceph 클러스터가 발급한 키를 Controller Node에 저장
```

---

## 6-6. Block Storage vs Shared Filesystem 네트워크 차이

```
Block Storage (Cinder)
→ Hypervisor가 VM과 스토리지 연결 처리
→ VM에서 별도 네트워크 구성 불필요

Shared Filesystem (Manila/CephFS)
→ VM이 직접 스토리지에 접근
→ provider-storage 네트워크 별도 필요
```

**VM 네트워크 구성**

```
VM
├── tenant network    ← 일반 통신용 (SSH, 인터넷)
└── provider-storage  ← 스토리지 전용 네트워크 (CephFS 접근용)
```

## 6-7. VM에 인증정보 주입 과정

```
1. Controller Node에서 keyring 생성
   /etc/ceph/ceph.client.cloud-user.keyring
        ↓
2. Controller Node → VM으로 keyring 파일 복사
   (Floating IP로 VM에 SSH 접속)
        ↓
3. VM의 cloud-user로 접속하여 인증파일 주입
        ↓
4. VM에서 CephFS 마운트 가능 ✅
```

```bash
# workstation에서 Controller Node의 keyring을 VM으로 복사
scp root@controller0:{cloud-user.keyring,/etc/ceph/ceph.conf} .

# VM 내부에서 CephFS 마운트
ceph-fuse /mnt/ceph/ \
  --id=cloud-user \
  --conf=/home/cloud-user/ceph.conf \
  --keyring=/home/cloud-user/cloud-user.keyring \
  --client-mountpoint=/volumes/_nogroup/<Share-UUID>
```
