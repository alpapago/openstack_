# 6. NFS

NFS구성의 controlplane역할을 하는게 manila 이고, 실제 데이터를 처리하고 저장 업무 진행하는놈이 NFS-Ganesha임
NFS Server로써 프로토콜을 구현해서 실제 저장역할을 하는게 Ganesha임
NFS 의 자원할당단위를 share라고 한다.

user가 요청 -> backend service에 구성된 worker들이 있음. 이를 manila-share가 관리하고있는데 DHSS: true일 때, 해당 manila share server를 띄움
-> NFS ganesha 배포하고 백엔드 스토리지 연결하고, share를 네트워크상에 share가 보이게끔 공유하는데 이를 export라고 한다. 

그런데 DHSS가 설정이 안되어 있으면 NFS-Ganesha가 배포가 안되고 manila-share가 구성되지 않고 별도의 NFS서버가 만들어지지 않고 백엔드 스토리지와 직접 통신한다.
이게 뢔 중요하냐면, 하나의 스토리지 서버를 쓸 때도 project마다의 보안이 중요함. 그래서 project마다 storage가 격리될 수 있도록 네트워크로 격리함 이 역할을 하는게 NFS Ganesha임.
Ganesha가 없으면, 백엔드 스토리지와 직접 통신하므로 격리가 되지 않음.
DHSS가 설정되어있어야 Ganesha가 배포 될 수 있으므로 manilar-share가 설치되기 전에 DHSS부터 설치 및 설정이 되어있어야 한다. Ganesha 를 LDAP서버랑 연결해서 인증작업을 진행할수도 있다.

ceph를 백엔드 스토리지로 사용하면서 nfs처리 요청을 할때는 cephx라는 프로토콜을 사용해야한다. 
인증 처리를 위해서 키값이 저장되는데 ~~~/keyring에 저장된다.

manila가 ceph에 접속할 때는 /etc/ceph/ceph.client.manila.keyring에 있는 키를 쓴대 맞나?
이게 저 keyring이 ceph에 있는건가? 어딧는거지?
FS은 provider-storage 라는 네트워크를 하나 더 붙여야 된대 그게 무슨말이지?
키링을 생성하면 vm에다가 인증정보를 넣어줘야함

key ring 을 controller node에서 생성한거임. cephFS의 요청에 의해, controller 노드에서 만든 키링파일을 접속할  vm으로 가져오고 이 파일을, cloud-user가 있는 vm의 floating ip를 가지고 접속해서 인증파일을 넣어준다. 



## 5-6. Shared Filesystem (Manila — NFS)

---

### Manila 구조

```
Control Plane: Manila
        ↓
Data Plane: NFS-Ganesha
        ↓
Backend Storage: Ceph 등
```

| 구성요소 | 역할 |
|---|---|
| **Manila** | NFS 구성의 Control Plane. 자원 할당/관리 담당 |
| **NFS-Ganesha** | NFS Server 역할. 실제 데이터 처리/저장 및 프로토콜 구현 |
| **Share** | NFS의 자원 할당 단위 |
| **Export** | Share를 네트워크상에 공유하는 것 |

---

### DHSS (Driver Handles Share Servers)

**DHSS: True (권장)**

```
사용자 요청
        ↓
Manila-share가 NFS-Ganesha 자동 배포
        ↓
백엔드 스토리지 연결
        ↓
Share를 네트워크에 Export
```

- Project마다 **NFS-Ganesha를 독립적으로 배포**
- Project 간 스토리지 **네트워크 격리** 가능
- DHSS가 설정되어야 Ganesha가 배포됨
- Manila Share Type 생성 시 DHSS 옵션 설정 필요

```bash
# DHSS true로 Share Type 생성
openstack share type create \
  --extra-specs driver_handles_share_servers=True \
  default_share_type
```

**DHSS: False**

```
NFS-Ganesha 배포 안됨
        ↓
백엔드 스토리지와 직접 통신
        ↓
Project 간 격리 안됨 ⚠️
```

---

### DHSS가 중요한 이유

```
하나의 스토리지 서버를 여러 Project가 공유할 때
→ Project 간 보안/격리가 필요
→ NFS-Ganesha가 네트워크 격리 역할 담당

DHSS: True  → Ganesha 배포 → Project별 격리 ✅
DHSS: False → Ganesha 없음 → Project별 격리 ❌
```

> Ganesha를 LDAP 서버와 연결하여 인증 작업을 진행할 수도 있음.

---

### Ceph 백엔드 연동

**cephx 프로토콜**
- Manila가 Ceph를 백엔드 스토리지로 사용할 때 **인증 프로토콜**
- 인증 키값은 **keyring 파일**에 저장됨

**Keyring 파일**

```
/etc/ceph/ceph.client.manila.keyring
→ Controller Node에 존재
→ Manila가 Ceph에 접속할 때 사용하는 인증 키
→ Ceph 클러스터가 발급한 키를 Controller Node에 저장
```

**provider-storage 네트워크**
- CephFS/Manila가 스토리지 백엔드와 통신하기 위한 **전용 스토리지 네트워크**
- VM이 Share에 접근하려면 이 네트워크를 추가로 붙여야 함

```
VM
├── tenant network    ← 일반 통신용
└── provider-storage  ← 스토리지 전용 네트워크 (CephFS 접근용)
```

---

### VM에 인증정보 주입 과정

```
1. Controller Node에서 keyring 생성
   /etc/ceph/ceph.client.manila.keyring
        ↓
2. 생성된 keyring 파일을 VM으로 복사
   (Floating IP로 VM에 SSH 접속)
        ↓
3. VM의 cloud-user로 접속하여 인증파일 주입
        ↓
4. VM에서 CephFS 마운트 가능 ✅
```

```bash
# Controller Node에서 keyring 복사
scp /etc/ceph/ceph.client.manila.keyring \
  cloud-user@:/home/cloud-user/

# VM 내부에서 인증파일 적용 후 마운트
mount -t ceph :/ /mnt \
  -o name=manila,secretfile=/home/cloud-user/ceph.client.manila.keyring
```
