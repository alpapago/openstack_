## 3. Software-defined Networking

### Underlay vs Overlay Network

| 구분 | 설명 | 예시 |
|---|---|---|
| **Underlay** | 실제 물리 네트워크 인프라 | 물리 스위치, 라우터, 케이블 |
| **Overlay** | Underlay 위에 가상화된 논리 네트워크 | VLAN, GENEVE 터널 |

OpenStack은 Project(Tenant) 단위로 수많은 고객의 네트워크를 독립적으로 제공해야 하므로,
물리 장비를 가상화하는 Overlay Network 기술이 필수적이다.

---

### Overlay 가상화 기술 — GENEVE

GENEVE(Generic Network Virtualization Encapsulation)는 VLAN보다 훨씬 유연하고 강력한 **터널링 방식**의 네트워크 가상화 기술이다.

**VLAN과의 차이**
| 구분 | VLAN | GENEVE |
|---|---|---|
| **계층** | L2 | L2 over UDP |
| **헤더** | 고정 길이 | **가변 길이** (메타데이터 확장 가능) |
| **전송 방식** | 이더넷 프레임 | **UDP 캡슐화** |
| **확장성** | 최대 4096개 | 사실상 무제한 |

> GENEVE는 기존 패킷을 UDP로 캡슐화하여 터널을 통해 전달하므로,
> 물리 네트워크 구조와 무관하게 논리 네트워크를 자유롭게 구성할 수 있다.

---

### 리소스 가상화 — SDx

| 기술 | 풀네임 | 가상화 대상 |
|---|---|---|
| **SDC** | Software Defined Compute | 서버/컴퓨팅 (→ Nova) |
| **SDS** | Software Defined Storage | 스토리지 (→ Cinder, Swift) |
| **SDN** | Software Defined Network | 네트워크 스위치/라우터 (→ Neutron) |

---

### OpenStack의 SDN 구현 — Neutron + OVN + OVS

OpenStack에서 네트워크 가상화는 **Neutron이 관리자 역할**을 하며,
내부적으로 **OVN(Control Plane)** 과 **OVS(Data Plane)** 이 실제 동작을 담당한다.

| 구분 | 역할 | 담당 |
|---|---|---|
| **Control Plane** | 네트워크 정책·라우팅 규칙 정의 | Neutron, OVN |
| **Data Plane** | 실제 패킷 포워딩 | OVS |

---

#### 패킷 처리 흐름

````
① 사용자 API 요청 (네트워크 생성/변경)
        ↓
② Neutron + OVN ML2 Plugin                     [Compute Node]
   사용자 요청을 수신하여 OVN Northbound DB로 전달
        ↓
③ OVN Northbound Database                      [Data Node]
   사용자 요청을 그대로(원시 데이터) 저장
   (ex. "A VM → B VM 통신 허용")
        ↓
④ ovn-northd                                   [Compute Node]
   Northbound DB를 읽어 논리 흐름(Logical Flow)으로 가공
   → "패킷이 어떤 규칙으로 어디에 전달되어야 하는지" 정의
        ↓
⑤ OVN Southbound Database                      [Data Node]
   논리 흐름 저장
   각 Compute Node의 ovn-controller가 DB를 Watch하다가
   변경 감지 시 즉시 해당 정보를 가져감
        ↓
⑥ ovn-controller → OpenFlow DB                [각 Compute Node]
   논리 흐름을 물리 네트워크에서 실행 가능한
   OpenFlow 규칙으로 변환하여 OpenFlow Table에 저장
        ↓
⑦ OVS Switch                                   [각 Compute Node]
   OpenFlow Table의 규칙을 보고 실제 패킷 포워딩 처리
````

---

#### OpenFlow

SDN의 핵심 기술로, 물리적인 라우터·스위치 등 네트워크 장비의
트래픽 처리 방식을 **소프트웨어로 프로그래밍**할 수 있게 해주는 프로토콜.
물리 장비와 관계없이 다이나믹하게 패킷 전달 방식을 변경할 수 있다.
논리 흐름 (OVN Southbound DB)
↓ 변환
ovn-controller
↓ OpenFlow 규칙 생성 및 적용
OVS Flow Table (OVS 내부에 저장)
↓ 참조
OVS Switch → 실제 패킷 포워딩

> ovn-controller가 Southbound DB의 논리 흐름을 읽어
> OpenFlow 규칙으로 변환하고, 이를 OVS의 Flow Table에 직접 주입(작성(한다.
> OVS Switch는 Flow Table을 참조하여 실제 패킷을 포워딩한다.

---

#### Overlay Network와 HAProxy

**Overlay Network**는 VM 간 통신 시 물리 네트워크 위에 **GENEVE 터널링**으로
논리적인 가상 네트워크를 구성하여 통신한다.

````
[VM A]                              [VM B]
  ↓                                   ↑
OVS                               OVS
  ↓                                   ↑
GENEVE 터널 (Underlay Network를 통해 캡슐화 전송)
````

**HAProxy**는 VM의 메타데이터 요청을 빠르게 처리하기 위해 사용된다.

- VM 부팅 시 내부적으로 `169.254.169.254` 메타데이터 API에 접근
- 이 요청을 **HAProxy가 로컬에서 프록시**하여 빠르게 응답
- Neutron의 metadata-agent와 연동하여 동작

````
VM 부팅
  ↓ 메타데이터 요청 (169.254.169.254)
HAProxy (로컬 프록시)
  ↓
Neutron Metadata Agent
  ↓
Nova Metadata API → VM 설정 정보 반환
````

#### Controller Node 구성요소

Controller Node는 하이퍼바이저 없이 **네트워크 제어만 담당**하는 노드.
실제 VM(Instance)은 없고, 외부 네트워크와의 연결을 처리한다.

**① OVS Integration Bridge (br-int)**
- OVS의 핵심 가상 스위치
- 모든 내부 트래픽이 이 브릿지를 통과함
- OpenFlow Table(switching, routing, access rules)을 보유

**② OVS Provider Bridge (br-ex)**
- 외부 물리 네트워크(eth2)와 연결되는 브릿지
- NIC eth2 → PORT eth2로 외부 트래픽 수신

**③ Patch Pair (bt-int-to-provnet / provnet-to-br-int)**
- br-int ↔ br-ex 를 연결하는 **논리적 가상 케이블**
- 단방향이 아닌 양방향 패치로 연결
- 논리 스위치(br-int)와 논리 라우터(br-ex) 역할을 연결해주는 다리
br-int  ←→  Patch Pair (bt-int-to-provnet)  ←→  br-ex
Patch Pair (provnet-to-br-int)

**④ GENEVE Patch (ovn-tunnel-ID)**
- eth1을 통해 Compute Node와 **GENEVE 터널링**으로 연결
- Overlay Network 통신 담당
- Controller Node ↔ Compute Node 간 VM 트래픽을 캡슐화하여 전달

---

#### Compute Node 구성요소

Compute Node는 실제 **VM(Instance)이 동작**하는 노드.

**① Instance (VM)**
- instance1, instance2 두 개의 VM이 동작
- 각 VM은 **TAP 디바이스(eth0)** 를 통해 br-int에 연결
- TAP: VM의 가상 NIC를 호스트의 OVS에 연결해주는 가상 인터페이스

**② OVS Integration Bridge (br-int)**
- VM의 TAP 포트를 PORT tap으로 수신
- OpenFlow Table로 switching/routing/access rules 처리
- GENEVE Patch를 통해 다른 노드와 Overlay 통신

**③ GENEVE Patch (ovn-tunnel-ID)**
- eth1을 통해 Controller Node 및 다른 Compute Node와 터널링
- **VM 간 Overlay 통신의 핵심 경로**

**④ Metadata Namespace (network1, network2)**
- 각 네트워크(VM)별로 독립된 메타데이터 네임스페이스 존재
- **HAProxy**가 각 네임스페이스 안에서 동작
- VM 부팅 시 `169.254.169.254` 메타데이터 요청을 로컬에서 빠르게 처리
VM 부팅 → 메타데이터 요청
↓
METADATA NAMESPACE
HAProxy        ← 네트워크별로 독립 동작
↓
Neutron Metadata Agent
↓
Nova Metadata API

---

#### 트래픽 흐름 정리

**① VM → 외부 인터넷 (Overlay → Underlay)**
instance1 (TAP eth0)
↓
br-int (OpenFlow 처리)
↓
GENEVE Patch (eth1) → SWITCH internal
↓
Controller Node br-int
↓
Patch Pair → br-ex (eth2)
↓
SWITCH external → 외부 인터넷

**② 같은 Compute Node 내 VM 간 통신**
instance1 (TAP)
↓
br-int OpenFlow Table에서 바로 처리
↓
instance2 (TAP)
→ GENEVE 터널 불필요, br-int 내부에서 직접 처리

**③ 다른 Compute Node의 VM 간 통신**
instance1
↓
br-int → GENEVE Patch (eth1)
↓ GENEVE 터널링 (Overlay)
다른 Compute Node의 eth1 → br-int
↓
대상 instance

**참고 — Compute Node vs Controller Node**

| 구분 | 하이퍼바이저 | 역할 |
|---|---|---|
| **Compute Node** | ✅ 있음 | VM(Instance) 실제 실행 |
| **Controller Node** | ❌ 없음 | 네트워크 제어, 외부 연결 처리 |

### Self-service Tenant Networking (Overlay)

프로젝트(Tenant) 내부 VM 간 통신을 위한 가상 네트워크.
물리 네트워크와 독립적으로 SDN 기반 터널링으로 구성된다.
VM내부 프로젝트의 네트워크는 VXLAN써서 쪼개놨음. VNI씀.
overlay에서 물리 네트워크로 통신하기위해서 SDN기반의 터널링쓸거고 터널링을 하면서 VNI로 가상화 해서 다른 VM에 있는 프로젝트로 간다.

**핵심 기술: GENEVE 터널링 + VNI**

| 용어 | 설명 |
|---|---|
| **GENEVE** | VM 간 Overlay 통신을 위한 터널링 프로토콜 |
| **VNI** (Virtual Network Identifier) | 터널 안에서 각 가상 네트워크를 구분하는 ID. VLAN의 VLAN ID와 유사한 개념 |
[Project A의 VM1]                    [Project A의 VM2]
↓                                      ↑
br-int                                 br-int
↓                                      ↑
GENEVE 터널 (VNI로 프로젝트 구분)————————————
↓
물리 네트워크(Underlay)를 통과하지만
VNI로 격리되어 다른 프로젝트와 섞이지 않음

> VNI가 다르면 같은 물리 네트워크를 공유해도
> 서로 다른 프로젝트의 트래픽은 완전히 격리된다.

---

### Provider Networking (Underlay)

물리 네트워크를 직접 활용하는 방식. Tenant Networking과 달리 터널링 없이
**물리 스위치(Hardware Switch)** 와 직접 연동한다.

**Provider Network 타입 비교**

| 타입 | 설명 | 특징 |
|---|---|---|
| **Flat** | VLAN 태그 없이 물리 네트워크를 그대로 사용 | 네트워크 분리 없음, 단순 구성 |
| **VLAN** | 물리 스위치의 VLAN 태그로 네트워크를 분리 | 하드웨어 스위치에서 VLAN ID로 격리 |

> Flat과 VLAN은 반댓말 관계가 아니라,
> **둘 다 Provider Network의 타입**이다.
> Flat은 "태그 없음", VLAN은 "태그 있음"의 차이.
Flat  → [VM] → br-ex → 물리NIC → 하드웨어 스위치 (태그 없음)
VLAN  → [VM] → br-ex → 물리NIC → 하드웨어 스위치 (VLAN ID 태그)

**Self-service vs Provider 비교**

| 구분 | Self-service (Tenant) | Provider |
|---|---|---|
| **네트워크 계층** | Overlay (가상) | Underlay (물리) |
| **터널링** | GENEVE + VNI | 없음 |
| **스위치** | OVS (소프트웨어) | Hardware Switch |
| **격리 방식** | VNI | VLAN ID |
| **관리 주체** | 테넌트(사용자) | 관리자(Admin) |


**참고**
하이퍼바이저위에 있다는건 compute node라는 뜻.
테넌트 끼리는 사설 ip를 쓰기 때문에 ip가 겹쳐도 상관 없음 안에서만 쓸거라서 dhcp로 자동할당되어도 상관없다.
porvider network는 public ip라서 겹치면 안되기 때문에 provider network의dhcp는 false되어있다.
