### Disk 구성 및 저장 구조

Flavor로 할당된 디스크는 **VM 내부 디스크가 아니라 Compute Node(하이퍼바이저)의 디스크**를 사용한다.
KVM/QEMU 가상화를 통해 VM 내부에서는 `/dev/vd*` 형태의 가상 디스크로 보인다.

```
Compute Node 로컬 (Ceph 미구성 시)
└── /var/lib/nova/instances/<instance-uuid>/
    ├── disk        ← Root Disk     → VM 내부: /dev/vda  (자동 포맷/마운트)
    ├── disk.local  ← Ephemeral     → VM 내부: /dev/vdb  (수동 포맷/마운트 필요)
    └── disk.swap   ← Swap Disk     → VM 내부: /dev/vdc

Ceph 구성 시
├── vms pool: Root Disk  → VM 내부: /dev/vda
├── vms pool: Ephemeral  → VM 내부: /dev/vdb
├── Compute Node 로컬: Swap Disk → VM 내부: /dev/vdc  ← Ceph 레이턴시 문제로 로컬 사용
└── volumes pool: Cinder Volume  → VM 내부: /dev/vdd~  (수동 포맷/마운트 필요)
```

---

#### Root Disk (루트 디스크)
- Boot 영역이 포함된 메인 디스크
- OS 설치 시 **자동으로 포맷/마운트** 완료
- 반드시 **이미지의 min_disk 크기보다 커야** 배포 가능

```
Flavor Root Disk 크기 > 이미지 min_disk 크기   ← 미충족 시 배포 불가
```

#### Ephemeral Disk (임시 디스크)
- 임시 작업용 추가 디스크 **(optional)**
- 각 VM의 임시 파일 저장 용도
- VM 삭제 시 데이터도 함께 삭제됨
- **수동으로 포맷/마운트** 해야 사용 가능

#### Swap Disk (스왑 디스크)
- **가상 메모리(Virtual Memory)** 기술
- RAM이 부족할 때 디스크 일부를 메모리처럼 활용하는 영역
- Ceph 구성 시에도 레이턴시 문제로 **항상 Compute Node 로컬**에 저장

```
Virtual Memory = RAM + Swap Disk
├── RAM        ← 빠름, 실제 메모리
└── Swap Disk  ← 느림, 디스크를 메모리처럼 사용
```

> Swap을 0으로 설정하면 RAM만 사용하는 구성이 되며,
> 메모리 초과 시 OOM(Out of Memory)으로 프로세스가 종료됨.
> 성능이 중요한 워크로드에서는 Swap을 비활성화하기도 함.

### Ephemeral Disk / Cinder Volume 수동 마운트 방법

Root Disk를 제외한 디스크는 attach 후 아래 순서로 직접 포맷/마운트해야 사용 가능하다.

```bash
# 1. 디바이스 확인
lsblk
# → /dev/vdb 확인

# 2. 파티션 생성
fdisk /dev/vdb        # MBR 방식 (2TB 이하)
parted /dev/vdb       # GPT 방식 (2TB 초과 대용량 디스크)

# 3. 파일시스템 포맷
mkfs.xfs /dev/vdb1    # XFS 포맷 (RHEL 권장)
mkfs.ext4 /dev/vdb1   # ext4 포맷

# 4. 마운트
mount /dev/vdb1 /data

# 5. 영구 마운트 설정 (재부팅 후에도 유지)
echo "/dev/vdb1 /data xfs defaults 0 0" >> /etc/fstab
```

**Ceph 유무에 따른 실제 저장 위치 요약**

| 디스크 | Ceph 미구성 | Ceph 구성 |
|---|---|---|
| **Root Disk** | Compute Node 로컬 | Ceph vms pool |
| **Ephemeral Disk** | Compute Node 로컬 | Ceph vms pool |
| **Swap Disk** | Compute Node 로컬 | Compute Node 로컬 (항상) |
| **Cinder Volume** | Cinder 백엔드 | Ceph volumes pool |

> VM 입장에서는 모두 똑같이 `/dev/vd*` 로 보이지만,
> 실제 데이터가 어디에 저장되느냐는 Ceph 구성 여부에 따라 결정된다.

## Glance 이미지 저장 방식 — Ceph 유무에 따른 차이

### openstack image create
로컬 이미지 파일을 **Glance에 등록**하는 명령어.
이미지를 새로 만드는 게 아니라, 이미 존재하는 이미지 파일을 Glance 백엔드에 업로드하는 것.

```
openstack image create --file rhel8.qcow2
        ↓
Glance가 이미지 수신
        ↓
┌─────────────────────────────────────────────────────┐
│ Ceph 미구성                                          │
│ Controller Node 로컬에 저장                          │
│ └── /var/lib/glance/images/<image-uuid>             │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│ Ceph 구성                                            │
│ → Ceph images pool에 저장                            │
│ → Glance는 메타정보만 관리                            │
│   (rbd://…/images/<image-uuid>/snap 형태로 참조)     │
└─────────────────────────────────────────────────────┘
```

> 어느 경우든 **Glance가 이미지 등록/관리의 주체**이며,
> Ceph 구성 시에는 Glance가 Ceph를 백엔드로 사용하는 것.
> Ceph를 다루는 CLI 도구는 **rbd**

### 1) Ceph 미구성 — 로컬 스토리지 방식

**저장 구조**

```
Controller Node
└── Glance /var/lib/glance/images/<image-uuid>   ← 이미지 원본 저장
        ↓ 네트워크를 통해 복사 (부하 발생)
Compute Node
└── /var/lib/nova/instances/<instance-uuid>/
    ├── disk        ← Root Disk (Base Image 복사본)
    ├── disk.local  ← Ephemeral Disk
    └── disk.swap   ← Swap Disk
```

**Base Image & Delta Disk(overlay disk)**
```
Base Image (읽기 전용, 여러 VM이 공유)
        ↓
각 VM마다 Delta Disk 보유
(Base Image와의 차이점만 저장)
```

- Base Image는 변경되지 않으며 여러 VM이 공유
- VM별 변경사항은 각자의 Delta Disk에 저장
- **Base Image + Delta Disk 합산 크기 < Flavor Disk 크기** 여야 배포 가능

**Flavor → 실제 저장 위치 (Ceph 미구성)**
```
Flavor 정의 (Root/Ephemeral/Swap Disk 크기)
        ↓
Compute Node 로컬(/var/lib/nova/instances)에 저장
├── /dev/vda ← Root Disk
├── /dev/vdb ← Ephemeral Disk
└── /dev/vdc ← Swap Disk
```

**장단점**
| 장점 | 단점 |
|---|---|
| 로컬 디스크 I/O라 이미지 읽기 속도 빠름 | 이미지 복사 시 네트워크 부하 발생 |
| 구성 단순 | Delta Disk 누적으로 로컬 디스크 공간 부족 위험 |
| | 장애 시 Failover 불가 (단일 장애점) |

---

### 2) Ceph 구성 — 분산 스토리지 방식

**저장 구조**
```
Ceph 클러스터
├── images pool   ← Glance: Base Image 원본 저장
├── vms pool      ← Nova: VM별 COW Delta(Overlay Disk) 저장 → VM과 생명주기 동일
└── volumes pool  ← Cinder: Block Volume 저장 → VM과 독립적, 영구 보존
```

- Glance는 이미지 원본 대신 **메타정보만 관리** (생성 시각, Ceph 내 저장 위치 등)
- Glance가 Nova에게 이미지 위치(메타정보) 전달
- Nova가 Ceph에 직접 접속하여 **COW(Copy-on-Write) 방식으로 내부 클론**
  - 네트워크로 복사하는 게 아니라 Ceph 내부에서 직접 클론 → 네트워크 부하 없음

**Flavor → 실제 저장 위치 (Ceph 구성)**
```
Flavor 정의 (Root/Ephemeral/Swap Disk 크기)
        ↓
┌─────────────────────────────────────┐
│ images pool                         │
│ └── Base Image (읽기 전용)           │
│         ↓ COW 클론                  │
│ vms pool                            │
│ ├── Root Disk     → /dev/vda        │
│ └── Ephemeral     → /dev/vdb        │
│                                     │
│ Compute Node 로컬                    │
│ └── Swap Disk     → /dev/vdc        │
│                                     │
│ volumes pool                        │
│ └── Cinder Volume → /dev/vdd~       │
└─────────────────────────────────────┘
```

> Swap Disk만 Compute Node 로컬에 저장되는 이유:
> Ceph 네트워크 스토리지를 Swap으로 사용하면 레이턴시가 너무 커서 사실상 사용 불가능하기 때문.

**장단점**
| 장점 | 단점 |
|---|---|
| COW 클론으로 빠른 VM 생성, 네트워크 부하 없음 | Ceph 구성 복잡도 높음 |
| 분산 스토리지로 디스크 공간 부족 위험 없음 | 초기 구축 비용 높음 |
| Failover 가능 (다른 host가 동일 Ceph를 바라보면 됨) | |
| Host 변경에도 공유 스토리지만 바라보면 되므로 유연한 운영 가능 | |

## 이미지 속성 (Image Visibility)

| 속성 | 설명 |
|---|---|
| **private** | 이미지를 생성한 사용자만 사용 가능 |
| **shared** | 같은 프로젝트 내 사용자 모두 사용 가능 **(default)** |
| **public** | 모든 프로젝트 사용자 접근 가능 (admin만 설정 가능) |
| **protected** | 삭제 불가 (실수로 인한 이미지 삭제 방지용) |
