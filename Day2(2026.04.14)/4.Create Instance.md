## Glance 이미지 저장 방식 — Ceph 유무에 따른 차이

### 1) Ceph 미구성 — 로컬 스토리지 방식

**저장 구조**
- Glance가 설치된 Controller Node의 로컬 디스크(`/var/lib/nova/instances`)에 이미지 저장
- VM 생성 시 Nova가 Glance로부터 base image를 Compute Node 로컬로 **네트워크를 통해 복사**

**Base Image & Delta Disk**
Base Image (읽기 전용, 여러 VM이 공유)
↓
각 VM마다 Delta Disk 보유
(Base Image와의 차이점만 저장)
- Base Image는 변경되지 않으며 여러 VM이 공유
- VM별 변경사항은 각자의 Delta Disk에 저장
- **Base Image + Delta Disk 합산 크기 < Flavor에 정의된 Disk 크기** 여야 배포 가능

**장단점**
| 장점 | 단점 |
|---|---|
| 로컬 디스크 I/O라 이미지 읽기 속도 빠름 | Compute Node로 이미지 복사 시 네트워크 부하 발생 |
| 구성 단순 | Delta Disk 누적으로 로컬 디스크 공간 부족 위험 |
| | 장애 시 Failover 불가 (단일 장애점) |

---

### 2) Ceph 구성 — 분산 스토리지 방식

**저장 구조**
Ceph 클러스터
├── images pool   ← Glance 이미지 저장 위치
└── vms pool      ← VM의 Delta(변경) 정보 저장 위치

- Glance는 이미지 원본 대신 **메타정보만 관리**
  - 예: 생성 시각, Ceph 내 저장 위치(images pool 경로) 등
- Glance가 Nova에게 이미지 위치(메타정보) 전달
- Nova가 Ceph에 직접 접속하여 **COW(Copy-on-Write) 방식으로 클론**
  - 네트워크로 이미지를 복사해오는 게 아니라 Ceph 내부에서 직접 클론
  - 네트워크 부하 없음, 속도 빠름

**Base Image & Delta**
Ceph images pool: Base Image (공유, 읽기 전용)
↓ COW 클론
Ceph vms pool: VM별 Delta 정보 저장

**장단점**
| 장점 | 단점 |
|---|---|
| 이미지 복사 없이 COW 클론으로 빠른 VM 생성 | Ceph 구성 복잡도 높음 |
| 분산 스토리지로 디스크 공간 부족 위험 없음 | 초기 구축 비용 높음 |
| Failover 가능 (장애 시 다른 host가 같은 Ceph를 바라보면 됨) | |
| Host가 바뀌어도 공유 스토리지만 바라보면 되므로 유연한 운영 가능 | |

## 이미지 속성 (Image Visibility)

| 속성 | 설명 |
|---|---|
| **private** | 이미지를 생성한 사용자만 사용 가능 |
| **shared** | 같은 프로젝트 내 사용자 모두 사용 가능 **(default)** |
| **public** | 모든 프로젝트의 사용자가 사용 가능 (admin만 설정 가능) |
| **protected** | 삭제 불가 (실수로 인한 이미지 삭제 방지용) |

## Flavor

---

- VM의 하드웨어 스펙(vCPU, RAM, Disk)을 정의하는 템플릿
- **생성/수정/삭제는 admin 권한** 필요
- **사용(VM 생성 시 선택)은 일반 유저도 가능**

## Flavor Disk 구성

### Root Disk (루트 디스크)
- Boot 영역이 포함된 메인 디스크
- 반드시 **이미지의 min_disk 크기보다 커야** 배포 가능
Flavor Root Disk 크기 > 이미지 min_disk 크기   ← 미충족 시 배포 불가

### Ephemeral Disk (임시 디스크)
- 임시 작업용 추가 디스크 **(optional)**
- 각 VM의 임시 파일 저장 용도
- VM 삭제 시 데이터도 함께 삭제됨

### Swap Disk (스왑 디스크)
- **가상 메모리(Virtual Memory)** 기술
- RAM이 부족할 때 디스크 일부를 메모리처럼 활용하는 영역
Virtual Memory = RAM + Swap Disk
├── RAM        ← 빠름, 실제 메모리
└── Swap Disk  ← 느림, 디스크를 메모리처럼 사용

> Swap을 0으로 설정하면 RAM만 사용하는 구성이 되며,
> 메모리 초과 시 OOM(Out of Memory)으로 프로세스가 종료됨.
> 성능이 중요한 워크로드에서는 Swap을 비활성화하기도 함.
