# Chapter 2. Multi-Tenant Cloud Project Management

## 2-1. Default Domain

Default Domain은 OpenStack 설치 시 **자동으로 생성**되는 기본 도메인.
추가 Domain 생성은 **CLI로만 가능** (Horizon 대시보드 미지원)

Default Domain 안에는 기본적으로 두 가지 프로젝트가 존재한다.

| 프로젝트 | 설명 |
|---|---|
| **admin** | 시스템 전체를 관리하는 superuser 계정이 소속. 클러스터 전체 자원에 대한 최고 권한 보유 |
| **service** | Nova, Cinder, Swift 등 OpenStack 서비스 컴포넌트들의 내부 서비스 계정 모음. 서비스 간 API 호출 시 사용 |

---

## 2-2. 인가 (Authorization) — Role 기반 접근 제어 (RBAC)

Role은 아래 3단계 계층 구조로 구성되며, **상위 Role 부여 시 하위 Role 권한이 자동 포함**된다.
admin > member > reader

| Role | 권한 범위 |
|---|---|
| **admin** | 프로젝트 내 모든 자원 생성·수정·삭제 및 관리 |
| **member** | 자원 생성·수정·삭제 가능 (관리 기능 제외) |
| **reader** | 자원 조회만 가능 (읽기 전용) |

---

## 2-3. Hierarchy Project (계층형 프로젝트)

Keystone v3부터 프로젝트를 **Parent-Child 계층 구조**로 생성 가능 (Hierarchical Multitenancy).

- Parent 프로젝트에 접근 권한이 있는 사용자는 **Child 프로젝트에도 자동으로 접근 가능**
- 조직 내 팀/부서 구조를 그대로 프로젝트 계층으로 표현할 수 있어 대규모 환경 관리에 유용
Default Domain
└── parent-project      ← 여기 접근 권한이 있으면
├── child-project-A ← 여기도 접근 가능
└── child-project-B ← 여기도 접근 가능

---

## 2-4. 외부 인증 연동 (LDAP / Active Directory)

LDAP, Active Directory 같은 외부 인증 프로토콜 연동 시,
Default Domain과 분리된 **별도 Domain을 생성**하여 매핑하는 것을 권장.

> Default Domain과 외부 인증 체계를 분리하면
> 인증 장애 발생 시 영향 범위를 최소화할 수 있음.

## 2-5. 리소스 삭제 시 주의사항

### 계정(User) 삭제
계정 삭제 시 해당 계정이 생성한 **리소스(VM, Volume 등)는 함께 삭제되지 않음.**
삭제된 계정의 리소스는 소유자가 없는 **고아 리소스(Orphaned Resource)** 가 될 수 있으므로,
반드시 아래 순서를 지킬 것.

해당 계정의 리소스 삭제 (VM, Volume, Network 등)
계정 삭제


---

### 프로젝트(Project) 삭제
프로젝트 삭제 시에도 내부 자원을 먼저 정리해야 하며,
계층 구조가 있는 경우 **Child → Parent 순서**로 삭제해야 한다.

> Parent가 남아 있는 Child의 네임스페이스 역할을 하므로,
> Child가 존재하는 상태에서 Parent를 먼저 삭제하면 에러가 발생하거나 삭제가 거부됨.
> (Kubernetes에서 내부 자원이 남아 있는 네임스페이스가 `Terminating` 상태에서 멈추는 것과 동일한 원리)

안전한 삭제 순서:

프로젝트 내 인스턴스(Nova), 볼륨(Cinder), 네트워크(Neutron) 삭제
프로젝트에 연결된 사용자 Role Assignment 제거
Child 프로젝트부터 순서대로 삭제
openstack project set --disable <프로젝트_이름>
openstack project delete <프로젝트_이름>


---

### 도메인(Domain) 삭제
도메인은 수많은 사용자와 프로젝트를 포함하는 최상위 단위이므로,
실수로 인한 전체 시스템 마비를 방지하기 위해 **반드시 `disable` 후 삭제**해야 한다.

안전한 삭제 순서:

도메인 내 모든 자원(인스턴스, 볼륨 등) 정리
openstack domain set --disable <도메인_이름>
openstack domain delete <도메인_이름>
