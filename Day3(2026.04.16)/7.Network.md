## 6. Network 구성

---

## 6-1. Provider Network

- 물리 네트워크와 직접 연결되는 **외부 네트워크**
- Public IP 사용하므로 **IP 충돌 방지** 필수
- **DHCP 비활성화** (관리자가 수동으로 IP 관리)
- admin 권한이 있어야 생성 가능

```
물리 네트워크 (Public IP)
        ↓
Provider Network (External)
        ↓
Router External Gateway로 사용
        ↓
Floating IP를 통해 VM과 NAT 연결
```

```bash
# 1. Provider Network 생성 (admin 권한 필요)
openstack network create \
  --external \
  --share \
  --provider-network-type flat \
  --provider-physical-network datacentre \
  provider-datacentre

# 2. Provider Subnet 생성
openstack subnet create \
  --subnet-range 172.25.250.0/24 \
  --gateway 172.25.250.254 \
  --dns-nameserver 172.25.250.254 \
  --allocation-pool start=172.25.250.101,end=172.25.250.189 \
  --no-dhcp \
  --network provider-datacentre \
  provider-subnet-172.25.250
```

> Provider Network + Subnet 생성은
> **External Gateway로 사용할 네트워크를 준비**하는 것.
> 실제 External Gateway 설정은 Router에서 별도로 진행.

**옵션 설명**

| 옵션 | 설명 |
|---|---|
| `--external` | 외부 네트워크로 설정 (Floating IP용) |
| `--share` | 모든 프로젝트에서 공유 가능 |
| `--provider-network-type flat` | VLAN 태그 없이 물리 네트워크 그대로 사용 |
| `--provider-physical-network datacentre` | 연결할 물리 네트워크 이름 |
| `--no-dhcp` | DHCP 비활성화 (Public IP 충돌 방지) |

## 6-2. Tenant Network (Internal Network)

- 프로젝트 내부에서만 사용하는 **사설 네트워크**
- **DHCP 활성화** (유동 IP 자동 할당)
- **Subnet을 생성해야 사설 IP 사용 가능**
- Controller Node의 **Neutron**이 가상 네트워크/서브넷/라우터를 API로 관리

```bash
# 1. Tenant Network 생성
openstack network create \
  production-network1

# 2. Subnet 생성
openstack subnet create \
  --network production-network1 \
  --subnet-range 192.168.1.0/24 \
  --gateway 192.168.1.1 \
  --dns-nameserver 172.25.250.254 \
  production-subnet1
```

---

## 6-3. Router — External Gateway & Tenant Network 연결

- Tenant Network에서 외부 인터넷으로 나가기 위한 **Gateway 역할**
- Neutron Router가 **NAT**를 통해 사설 IP → 공인 IP 변환

```bash
# 1. Router 생성
openstack router create production-router1

# 2. Router에 Internal Subnet 연결
openstack router add subnet \
  production-router1 \
  production-subnet1
→ 내부 사설 IP를 Router에 연결

# 3. Router에 External Gateway 설정
openstack router set \
  --external-gateway provider-datacentre \
  production-router1
→ Tenant Network의 NAT가 External Network로 라우팅됨
```

**연결 구조**

```
provider-datacentre (외부 네트워크)
        ↑ External Gateway
production-router1 (NAT 처리)
        ↓ router add subnet
production-subnet1 (사설 IP 대역)
        ↓
VM들
```

## 6-4. Floating IP

- VM에 **공인 IP를 동적으로 할당**하는 기술
- Tenant Network의 사설 IP와 공인 IP를 1:1 NAT 매핑

**Floating IP 미리 연결해두면 장점**

```
VM이 재시작/재생성되어도 동일한 공인 IP 유지
→ DNS 설정 변경 불필요
→ 외부 서비스 연결 끊김 없음
→ 다른 VM으로 Floating IP 이전 가능 (장애 대응)
```

```bash
# 1. Floating IP 생성
openstack floating ip create provider-datacentre

# 2. VM에 Floating IP 할당
openstack server add floating ip \
  production-server9 \
  172.25.250.103
```

## 6-5. 가상 라우터 동작 방식 (OVN 기반)

### OVN DVR (Distributed Virtual Router)

```
OVN 기반에서는 가상 라우터가
Controller Node가 아닌 각 Compute Node에 분산 배포됨
→ OVS에 기록된 OpenFlow 라우팅 테이블을 보고 패킷 포워딩
→ Controller Node 부하 최소화
```

---

### North-South vs East-West

| 구분 | 트래픽 방향 | Controller Node 경유 |
|---|---|---|
| **East-West** | VM ↔ VM (내부 통신) | ❌ 불경유 |
| **North-South** | VM ↔ 외부 인터넷 | ✅ 경유 |

---

### 트래픽 흐름

**East-West (VM 간 내부 통신)**

```
같은 Compute Node 내 VM 간
VM1 → OVS br-int → VM2
→ OVS Flow Table에서 직접 처리
→ Controller Node 불경유 ✅

다른 Compute Node VM 간
VM1 → OVS → GENEVE 터널 → OVS → VM2
→ Controller Node 불경유 ✅
→ OVS OpenFlow 라우팅 테이블만 참조
```

**North-South (외부 인터넷 통신)**

```
VM → OVS → GENEVE → Controller Node → br-ex → 외부
→ Controller Node 경유 ⚠️
→ NAT(Floating IP) 처리도 Controller Node에서 수행
```

---

### 왜 North-South만 Controller Node를 거치냐?

```
Controller Node
└── br-ex (외부 네트워크 연결 포트)
    → 외부로 나가려면 반드시 br-ex 통과
    → br-ex는 Controller Node에만 존재
    → Floating IP NAT 처리도 여기서 수행
```
