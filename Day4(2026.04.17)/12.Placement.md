# 12. Placement Service

### 자원 배치 최적화가 중요함.

리소스 프로바이더가 자원을 제공하는데 인벤토리를 추적하면 자원 할당량 등을 알 수 있음
자원에 대한 사용량을 추적하여서 배치에 도움을 주는게 배치 서비스임.
리소스 단위를 표준화 해놓은게 클래스인데 클래스 단위로 리소스 추적함.
각 서비스들은 자체 db가 있어서 거기에 가용 cpu, 가용 메모리, 메모리 사용량 같은것들을 배치 서비스 자체 스토리지에 저장한다.
배치 서비스를 쿼리 하려면 python3-osc-placement패키지를 설치해야함.

리소스추적

리소스프로바이터 : 컴퓨트노드 컨트롤러노드,스토리지노드 등 노드들이 이에 해당함. 클래스 인벤토리 제공하는놈.
리소스 클래스 : 표준화된 리소스 타입. 커스텀해서 사용할 수 있음.

인벤토리: 가용자원 및 사용자원 정보를 의미함.
리소스 프로바이더 집계 : 해당 리소스를 제공하는 host들이 어느 node 에 있고 어느 host aggregation등을 확인할 수있다..?

정성적 리소스특징(trait)을 이용한 관리방식: 리소스 특성 종류 품질 유형 등에 의해 관리하는것 비표준적인 정의이고 사용자 지정 정의임. 
얘: vm 배포하는데 ssd사용해서 하고싶다~
지정 가능한 리소스 정의가 뭐가 있는지 확인할 때는 train을 확인하는 openstack resource provider trati ~ 으로 확인하면된다.
이런 정성적 특징을 image에 지정할 수도있다.

호스트 집계: 호스트의 집계구성 가능한 메타데이터 기반으로 hypervisor host를 그룹화 할 수 있다.

배치집계:모든서비스대상으로 가용자원등을 집계함.


배치 서비스가 중요한데... 

VM생성 요청을 하면 nova api를 통해서 요청이 간다. nova api 내부처리 유형이 여러가지가 잇음.
nova conductor가 잇고,얘가 요청을 받아서 nnova scheduler로 요청을 보냄. 그리고 얘가 placement 서비스한테 어디에 배치하면 좋을지 물어본다.
placement가 사용가능한 자원을 inventory에 조회한다. 그리고 이미지의 flavor정보를 확인해서 배치 가능한 후보 host목록을 scheduler에게 보내준다. 
후보 목록에서 filtering을 nova scheduler가 진행한다. 기준은 AZ , host 집계 단위 (예를 들면 SSD), 지리적으로 걸러놓고 남는애들 중에 상대적으로 한가한 워크로드나 cpu affinity설정한 것들을 고려해서  가중치 설정. 
최종 host선택하고 배포를 위해서 자원을 선점(claim)한다. 선점 했다고 placement가 nova scheduler에 return을 해주면, subnit해서 배포한다.

VM 생성요청 > nova api > nova conductor > nova scheduler > placement


# 12. Placement Service

---

## 12-1. Placement Service란?

**자원 배치 최적화**를 위해 각 노드의 자원 사용량을 추적하고
VM 배치 시 최적의 호스트를 찾아주는 서비스.

- Placement 전용 **독립 DB**에 자원 정보 저장
- 쿼리하려면 **python3-osc-placement** 패키지 설치 필요

```bash
pip install python3-osc-placement
```

---

## 12-2. 주요 개념

### Resource Provider (리소스 프로바이더)
- 자원을 제공하는 엔터티
- Compute Node, Controller Node, Storage Node 등이 해당
- 리소스 클래스의 **인벤토리를 제공**하는 주체

### Resource Class (리소스 클래스)
- **표준화된 리소스 타입**
- 커스텀해서 사용 가능

| 기본 리소스 클래스 | 설명 |
|---|---|
| `VCPU` | 가상 CPU |
| `MEMORY_MB` | 메모리 (MB) |
| `DISK_GB` | 디스크 (GB) |

### Inventory (인벤토리)
- 각 Resource Provider의 **가용 자원 및 사용 자원 정보**
- 예: 전체 vCPU 40개 중 사용 중 10개, 가용 30개

### Trait (정성적 리소스 특성)
- 비표준적/사용자 정의 리소스 특성
- 품질, 유형 등 **정성적 특징**으로 자원 관리
- Image나 Flavor에 지정 가능

```bash
# 사용 가능한 Trait 확인
openstack resource provider trait list

# 예시: SSD 스토리지를 사용하는 VM 배포 시
# Image에 trait 지정
openstack image set \
  --property trait:STORAGE_DISK_SSD=required \
  rhel8
```

### Host Aggregation (호스트 집계)
- 메타데이터 기반으로 **Hypervisor Host를 그룹화**
- 성능/기능 기준으로 그루핑 (장소 기준 아님)

### Placement Aggregate (배치 집계)
- 모든 서비스 대상으로 **가용 자원을 집계**
- Host Aggregation과 연동하여 배치에 활용

## 12-3. VM 생성 시 배치 흐름

```
① 사용자 VM 생성 요청
        ↓
② Nova API 수신
        ↓
③ Nova Conductor
   요청을 Nova Scheduler로 전달
        ↓
④ Nova Scheduler
   Placement Service에 배치 가능한 호스트 목록 요청
        ↓
⑤ Placement Service
   Inventory 조회
   → Flavor/Image 스펙과 비교
   → 배치 가능한 후보 Host 목록 반환
        ↓
⑥ Nova Scheduler - Filtering
   후보 목록에서 아래 기준으로 필터링
   ├── AZ (Availability Zone)
   ├── Host Aggregation (예: SSD 그룹)
   └── 지리적 조건
        ↓
⑦ Nova Scheduler - Weighting (가중치)
   필터링된 후보 중 아래 기준으로 최적 Host 선택
   ├── 워크로드가 적은 Host
   └── CPU Affinity 설정 고려
        ↓
⑧ 최종 Host 선택 후 자원 선점 (Claim)
   → Placement가 자원 선점 확인 후 Nova Scheduler에 반환
        ↓
⑨ Nova Compute에 배포 요청 (Submit)
        ↓
⑩ VM 생성 완료
```

---

## 12-4. 흐름 요약

```
VM 생성 요청
        ↓
Nova API → Nova Conductor → Nova Scheduler
        ↓
Placement (후보 Host 목록 반환)
        ↓
Nova Scheduler (Filtering → Weighting → Host 선택)
        ↓
자원 선점(Claim) → 배포(Submit)
        ↓
VM 생성 완료
```
